# Android와 iOS의 기능 차이

## 목적

이 문서는 Flutter로 하나의 앱을 개발하더라도 Android와 iOS에서 출결 처리 기능이
달라지는 이유와 현재 지원 범위를 기록한다.

기능 차이는 Flutter가 공통 UI를 만들지 못해서가 아니다. Google 인증의 소유권,
브라우저에서 앱으로 돌아오는 링크 설정, 각 운영체제의 백그라운드 정책이 결합되어
발생한다.

## 공통으로 구현할 수 있는 기능

다음 기능은 Flutter 코드와 각 플랫폼용 알림 플러그인을 통해 두 플랫폼에서 공통으로
제공할 수 있다.

- 사용자 정보 설정과 로컬 저장
- 요일 반복 및 날짜 지정 일정 관리
- 공휴일 제외 계산
- 로컬 알림 예약과 알림 탭 처리
- 외부 브라우저에서 SKALA Google 인증 시작

다만 알림이 정확히 같은 시각에 표시되는지는 사용자의 권한 설정, 절전 정책 및 OS의
스케줄링 판단에 영향을 받는다.

## Google 계정 정보를 앱이 직접 사용할 수 없는 이유

휴대폰의 시스템 계정 목록에 Google 계정이 등록되어 있어도 앱이 해당 계정의
비밀번호, Google 로그인 쿠키 또는 다른 브라우저의 세션을 가져올 수는 없다. 계정
선택 화면과 자격 증명 처리는 Google 및 운영체제가 소유한 인증 화면에서 이루어진다.

SKALA 출결 API는 일반 Google ID 토큰만 받는 구조가 아니다. SKALA의 사전 확인과
Google OAuth를 완료한 뒤 발급되는 당일 출결용 Bearer 토큰을 요구한다. 따라서 앱은
계정을 저장해 다음 로그인 때 몰래 선택하거나 Google 로그인 요청을 자체 API 호출로
대체하지 않는다.

현재 인증 흐름은 다음과 같다.

```text
앱에서 SKALA 사전 확인
  → 시스템 외부 브라우저에서 Google OAuth
  → SKALA가 당일 출결 토큰 발급
  → att.skala-ai.com 콜백
  → 지원되는 플랫폼에서 앱으로 복귀
  → 상태 조회 및 사용자 승인에 따른 출결 요청
```

Google 비밀번호와 브라우저 쿠키는 앱에 저장하지 않으며, 출결 토큰도 영구 저장하지
않는다.

## 현재 Android 구현

Android에서는 다음 플랫폼 구성이 구현되어 있다.

- 네이티브 채널을 통해 Chrome Custom Tab으로 인증 URL 열기
- `https://att.skala-ai.com/...` 링크를 앱이 수신하도록 Intent Filter 등록
- 인증 콜백의 토큰을 사용자 지역·반·날짜·만료 정보와 대조
- 당일 출결 상태 조회
- 현재 상태에서 허용되는 입실·퇴실·외출·복귀 요청 전송
- 요청 후 상태를 다시 조회하여 실제 반영 여부 확인
- 정확 알람 권한이 허용된 경우 화면 잠금 중에도 예약 시각에 로컬 알림 표시

알림 자체가 백그라운드에서 출결 요청을 보내는 것은 아니다. 사용자가 알림을 눌러
외부 브라우저 인증을 완료해야 하며, 위험한 동작에는 앱의 확인 절차가 적용된다.

## 현재 iOS 구현

iOS에서도 일정 관리, 공휴일 제외 및 로컬 알림 예약 코드는 포함되어 있다. Google
인증은 Safari 등 외부 브라우저로 열 수 있다.

그러나 현재 프로젝트에는 `att.skala-ai.com`을 이 앱의 Universal Link로 검증하기
위한 Associated Domains 설정과 해당 도메인의 `apple-app-site-association` 연결이
구성되어 있지 않다. 이 도메인 서버 설정은 도메인 운영자의 협조가 필요하다.

그 결과 현재 iOS에서는 인증 후 콜백을 앱이 신뢰할 수 있는 방식으로 수신하여 출결
토큰을 넘겨받는 흐름이 완성되지 않았다. 앱은 Safari에서 SKALA 웹 화면을 열고
사용자가 웹에서 직접 원하는 출결 동작을 수행하도록 안내한다.

이는 iOS에서 출결 API 호출이 원천적으로 불가능하다는 뜻은 아니다. 올바른 Universal
Link 또는 SKALA가 허용하는 별도 OAuth 콜백 방식, iOS 실기기 검증이 마련되면 기능
범위를 확대할 수 있다.

## 백그라운드 실행 차이

Android와 iOS 모두 일반 앱이 임의의 시각에 깨어나 브라우저 로그인을 대신하고
네트워크 요청을 반드시 실행하도록 보장하지 않는다. 특히 사용자 상호작용이 필요한
OAuth는 화면이 잠긴 상태에서 자동으로 완료할 수 없다.

Android는 정확 알람, 알림 수신기 및 앱 링크를 이용해 예약 알림에서 인증 흐름으로
이어지는 구성이 비교적 직접적이다. 하지만 권한 미승인, 제조사 절전 정책 및 강제
종료 상태에 따라 제약이 생길 수 있다.

iOS는 로컬 알림 표시는 지원하지만 임의 시각의 지속적인 백그라운드 실행과 즉시
네트워크 요청을 앱이 예약하는 것을 보장하지 않는다. Background Tasks 역시 실행
시각을 시스템이 결정하므로 정시 출결 처리 장치로 사용할 수 없다.

따라서 현재 앱은 두 플랫폼 모두에서 다음 원칙을 사용한다.

- OS에는 출결 API 실행이 아니라 로컬 알림을 예약한다.
- 사용자가 알림을 눌러 인증을 시작한다.
- 인증이 끝난 뒤 서버 상태를 다시 확인한다.
- 현재 상태에서 허용되는 동작만 전송한다.
- 플랫폼이 신뢰할 수 있는 콜백을 제공하지 못하면 웹에서 수동 처리한다.

## 기능 비교

| 기능 | Android | iOS 현재 구현 |
|---|---|---|
| 사용자 정보 및 일정 관리 | 지원 | 지원 |
| 공휴일 제외 | 지원 | 지원 |
| 로컬 일정 알림 | 지원 | 지원 |
| 외부 브라우저 Google 인증 시작 | 지원 | 지원 |
| 인증 후 앱 콜백 | App Link로 구현 | Universal Link 미구성 |
| 앱에서 당일 상태 조회 | 지원 | 미지원 |
| 앱에서 출결 이벤트 전송 | 지원 | 미지원 |
| 화면 잠금 중 무인 OAuth·출결 실행 | 미지원 | 미지원 |

## 향후 iOS 지원 조건

다음 작업이 완료되면 iOS의 상태 조회와 출결 요청 지원을 재검토한다.

1. SKALA가 허용하는 iOS 콜백 URI 확정
2. Universal Link를 사용할 경우 도메인의 `apple-app-site-association` 제공
3. 앱의 Associated Domains 및 서명 설정
4. Safari 인증 후 앱 복귀와 토큰 전달에 대한 iOS 실기기 검증
5. 토큰 검증, 상태 조회 및 동작 전송에 대한 회귀 테스트

이 조건이 갖춰져도 Google 계정 자동 선택이나 화면 잠금 상태의 완전 무인 인증은
지원 대상으로 보지 않는다.
