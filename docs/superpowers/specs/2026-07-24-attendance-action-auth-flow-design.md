# 동작 중심 출결 인증과 당일 상태 캐시 설계

## 배경

현재 홈은 Google 인증 전에는 `출결 정보 확인하기` 카드를, 인증과 상태 조회가 끝난
뒤에는 `오늘 출결` 카드를 표시한다. 이 구조에서는 사용자가 상태를 보기 위해 먼저
인증 전용 화면 상태를 거쳐야 하고, 실제로 원하는 입실·퇴실·외출·복귀 동작과 인증
버튼이 분리되어 있다.

새 흐름은 인증을 독립적인 목적이 아니라 상태 조회나 출결 동작을 수행하기 위한
중간 단계로 취급한다. 홈에는 항상 오늘 출결 카드를 표시하고 사용자가 새로고침 또는
출결 동작을 선택했을 때만 필요한 인증을 진행한다.

이 문서는 구현 시
`2026-07-24-attendance-status-ux-design.md`의 인증 카드 전환과 프로세스 재시작 시
상태 폐기 부분을 대체한다. 기존 2×2 타일, 표시 형식과 완료 피드백 원칙은 유지한다.

이번 변경의 제품·실기기 검증 대상은 Android다. iOS는 자동 콜백이 완성되지 않았고
현재 신규 개발과 배포가 중단된 상태이므로 Android와 동일한 자동 처리 지원을
의미하지 않는다. 공용 화면을 변경할 때도 이 플랫폼 차이를 문서와 코드에서 명시한다.

## 목표

- 인증 여부와 관계없이 홈에 오늘 출결 카드를 항상 표시한다.
- 오늘 상태를 아직 조회하지 않았다면 실제 기록 부재와 구분되는 `확인 전`을 표시한다.
- 카드의 새로고침으로 출결 요청 없이 현재 상태만 조회할 수 있게 한다.
- 입실·퇴실·외출·복귀 버튼에서 필요한 인증과 동작 처리를 한 흐름으로 연결한다.
- 오늘 조회한 4개 상태를 앱 재실행 후에도 표시하되 토큰과 인증정보는 저장하지 않는다.
- 모든 출결 동작 직전에 서버 최신 상태를 조회해 저장된 표시 상태를 실행 근거로
  사용하지 않는다.
- 한국 날짜 변경, 사용자 변경 및 오래된 비동기 결과가 전날 상태를 되살리지 못하게
  한다.

## 제외 범위

- Google 비밀번호, 쿠키 또는 출결 토큰 저장
- 앱 시작과 동시에 자동으로 브라우저 인증 실행
- 사용자가 누르지 않은 출결 동작의 자동 전송
- SKALA API 주소와 서버 출결 순서 규칙 변경
- 알림 예약 및 향후 네이티브 알람 구현
- iOS 인증 콜백 완성 또는 Android와 동일한 iOS 자동 출결 처리

## 화면 구조

기존 `출결 정보 확인하기` 카드를 제거하고 홈 카드 순서를 다음처럼 고정한다.

```text
사용자 인사
사용자 정보
오늘 출결
오늘 예정된 동작
Google 인증 보안 안내
```

### 카드 머리글

카드 왼쪽에는 현재 한국 날짜와 요일을 포함한 제목을 표시한다.

```text
오늘 출결 · 7월 24일(금)
```

오른쪽에는 항상 새로고침 아이콘을 둔다. 접근성 이름은
`출결 상태 새로고침`으로 지정한다. 조회 중에는 같은 자리에 작은 진행 표시를
보여주고 중복 새로고침과 출결 동작을 비활성화한다.

새로고침은 상태 조회 전용이다. 이 동작으로 출결 요청을 전송하지 않는다.

### 상태 타일

기존 반응형 2×2 타일과 순서를 유지한다.

```text
입실              퇴실
확인 전            확인 전

외출              복귀
확인 전            확인 전
```

표시값은 다음 세 상태를 구분한다.

- 오늘 서버 상태를 아직 조회하지 않음: `확인 전`
- 오늘 서버 조회에서 실제 값이 없음: `없음`
- 서버 조회 값이 있음: 한국 시각 `HH:mm`

일반 휴대폰은 2×2, 좁은 폭이나 큰 접근성 글자 크기에서는 1열로 전환하는 현재
동작을 유지한다.

### 가능한 동작

오늘 상태가 `확인 전`이면 입실·퇴실·외출·복귀 버튼을 모두 표시한다. 이 버튼은
출결을 즉시 전송하는 버튼이 아니라 사용자의 처리 의도를 지정하는 시작점이다.

오늘 상태를 한 번이라도 조회했다면 저장된 4개 시각을 기준으로 순서상 가능한
동작만 표시한다. 로컬에 저장하지 않는 네트워크 허용 여부는 버튼 표시 근거로
사용하지 않는다. 실제 실행 가능 여부는 버튼을 누른 뒤 서버의 최신 상태와 현재
네트워크 판정으로 다시 확인한다.

## 저장되는 당일 상태

새로운 출결 상태 저장소는 다음 값만 로컬에 저장한다.

- 조회 결과가 속한 한국 날짜
- 마지막 성공 조회 시각
- 입실 시각
- 퇴실 시각
- 외출 시각
- 복귀 시각

Google 토큰, 브라우저 쿠키, 비밀번호와 `networkAllowed`는 저장하지 않는다.
서버 응답 원문 중 화면과 순서 판정에 필요한 4개 시각만 보존한다.

앱 시작 시 저장 날짜가 현재 한국 날짜와 같으면 표시 상태를 복원한다. 날짜가
다르거나 저장 형식이 손상됐으면 저장값을 삭제하고 `확인 전`으로 시작한다.
사용자 이름·캠퍼스·반이 변경될 때도 기존 당일 상태를 삭제한다.

저장된 상태는 화면 표시와 최초 버튼 정리에만 사용한다. 출결 요청 허가, 완료 판정
또는 네트워크 허용 판정에는 사용하지 않는다.

## 인증 목적과 메모리 토큰

앱은 사용자가 시작한 목적을 다음 중 하나로 메모리에만 기억한다.

- 상태 새로고침
- 입실
- 퇴실
- 외출
- 복귀
- 기존 일정 알림에서 시작한 출결 동작

오늘 유효한 토큰이 프로세스 메모리에 있으면 다시 브라우저를 열지 않고 재사용한다.
앱 재실행, 날짜 변경, 사용자 변경 또는 토큰 만료로 메모리 토큰이 없으면 Google
인증을 시작한다. 토큰은 계속 디스크에 저장하지 않는다.

진행 중 목적은 앱이 종료되면 복원하지 않는다. 앱이 다시 시작되면 사용자가
새로고침이나 동작을 다시 선택해야 한다.

## 실행 흐름

### 새로고침

```text
새로고침 선택
→ 오늘 메모리 토큰 검사
→ 유효한 토큰이 없으면 Google 인증
→ 인증 콜백으로 앱 복귀
→ 오늘 서버 상태 조회
→ 카드와 로컬 당일 상태 갱신
```

이 흐름에서는 어떤 경우에도 출결 요청을 보내지 않는다.

### 출결 동작

```text
입실·퇴실·외출·복귀 선택
→ 요청한 동작을 메모리에 기억
→ 오늘 메모리 토큰 검사
→ 유효한 토큰이 없으면 Google 인증
→ 오늘 서버 상태 조회
→ 요청 동작의 현재 실행 가능 여부 확인
→ 기존 동작별 확인 정책 적용
→ 출결 요청
→ 오늘 상태 재조회
→ 서버 반영 확인
→ 카드와 로컬 당일 상태 갱신
```

입실과 복귀는 현재 정책대로 추가 확인 없이 진행한다. 외출은 항상 확인하고,
퇴실은 기존 조기 퇴실 기준에 따라 확인한다.

최신 서버 상태에서 요청한 동작이 이미 처리됐거나 현재 순서상 불가능하면 출결
요청을 보내지 않는다. 최신 상태를 카드와 저장소에 반영하고 실행하지 않은 이유를
사용자에게 안내한다.

### 일정 알림

일정 알림에서 시작한 기존 인증 흐름도 같은 목적 모델을 사용한다. 일정 ID와 예정
발생 시각 검증, 삭제·변경된 일정 차단, 완료 기록 원칙은 유지한다. 알림 동작 역시
인증 후 최신 서버 상태를 확인한 경우에만 출결 요청 단계로 진행한다.

## 상태와 책임

### 프레젠테이션

- 오늘 출결 카드를 항상 렌더링한다.
- `확인 전`, `없음`, 실제 시각을 구분해 표시한다.
- 카드 머리글의 새로고침과 가능한 동작 버튼을 제공한다.
- 확인 대화상자, 완료 타일 강조와 스낵바를 계속 담당한다.
- 조회·인증·동작 진행 중에도 마지막 성공 상태 타일을 지우지 않는다.
- 오류를 별도 인증 카드로 전환하지 않고 오늘 출결 카드 안에 표시한다.

### `AttendanceController`

- 오늘 표시 상태, 메모리 토큰, 진행 목적과 비동기 작업 세대를 관리한다.
- 새로고침과 출결 동작 요청이 토큰 재사용 또는 브라우저 인증으로 이어지게 한다.
- 인증 콜백 이후 진행 목적에 맞게 조회만 끝내거나 출결 가능 여부 확인으로 이어간다.
- 모든 성공 조회를 출결 상태 저장소에 반영한다.
- 날짜·사용자 변경 시 토큰, 진행 목적, 표시 상태와 저장 상태를 함께 초기화한다.
- 저장 상태를 출결 전송 근거로 사용하지 않는다.

### 출결 상태 저장소

- 당일 날짜, 마지막 성공 조회 시각과 4개 출결 시각만 직렬화한다.
- 다른 날짜나 손상된 값을 불러오지 않고 정리한다.
- 토큰과 네트워크 판정은 받거나 저장하지 않는다.

### API 계층

- 현재 토큰의 날짜·만료·사용자 일치 검증을 재사용한다.
- 서버가 인증 만료를 명시하면 일반 서버 오류와 구분해 컨트롤러가 토큰을 지우고
  인증을 다시 시작할 수 있게 한다.
- 출결 요청 및 요청 후 상태 재조회 방식은 유지한다.

## 날짜와 동시성

한국 날짜 비교는 다음 시점에 모두 수행한다.

- 앱 시작과 저장 상태 복원
- 앱이 포그라운드로 복귀할 때
- 자정 만료 타이머
- 새로고침 또는 출결 동작 시작 직전
- 인증 콜백과 서버 요청 완료 직후

날짜가 바뀌면 토큰과 진행 목적을 폐기하고 저장된 상태를 삭제하며 타일을
`확인 전`으로 되돌린다. 전날 시작한 인증·조회·출결 요청의 늦은 결과는 작업 세대와
날짜 검사를 통과하지 못하므로 화면이나 저장소를 다시 채우지 못한다.

겹친 요청은 한 번에 하나만 실행한다. 진행 중에는 새로고침과 동작 버튼을
비활성화해 사용자가 다른 목적을 덮어쓰지 못하게 한다.

## 오류 처리

- 인증 취소·실패: 기존 타일을 유지하고 카드 안에 인증 실패와 재시도 안내 표시
- 조회 실패: 기존 타일과 로컬 상태를 유지하며 실패 결과로 저장값을 덮어쓰지 않음
- 토큰 만료: 메모리 토큰을 폐기하고 진행 목적을 유지한 채 인증을 한 번 다시 시작
- 네트워크 불허: 최신 조회 결과의 경고를 표시하고 출결 요청은 보내지 않음
- 동작 불가: 최신 상태를 먼저 반영하고 요청을 전송하지 않은 이유 표시
- 출결 결과 불명확: 기존 원칙대로 동작을 재전송하지 않고 상태만 다시 조회
- 출결 반영 미확인: 완료 강조와 완료 스낵바를 표시하지 않음
- 로컬 저장 실패: 서버에서 받은 메모리 상태는 표시하되 앱 동작을 중단하지 않음

오류 후 재시도는 원래 목적을 보존한다. 다만 출결 요청을 이미 보냈는데 결과가
불명확한 경우에는 원래 동작을 다시 보내지 않고 상태 조회만 허용한다.

## 검증

자동 테스트는 다음을 확인한다.

- 인증 전에도 오늘 출결 카드 하나만 표시되고 인증 카드는 나타나지 않는지
- 조회 전 4개 타일이 `확인 전`이고 실제 조회 후 값 없음이 `없음`인지
- 새로고침이 카드 우측 상단에 있으며 접근성 이름을 제공하는지
- 새로고침은 인증과 조회만 수행하고 출결 요청을 보내지 않는지
- 조회 전에는 4개 동작을 모두 제공하고 조회 후에는 순서상 가능한 동작만 표시하는지
- 오늘 메모리 토큰이 있으면 브라우저 인증 없이 조회·동작 흐름을 이어가는지
- 토큰이 없거나 만료되면 진행 목적을 유지한 채 인증 후 이어가는지
- 출결 동작 전에 항상 최신 서버 상태를 조회하고 불가능한 동작을 보내지 않는지
- 성공 조회가 당일 상태를 저장하고 앱 재생성 후 복원되는지
- 토큰과 `networkAllowed`가 로컬 저장에 포함되지 않는지
- 날짜 또는 사용자 변경 시 저장 상태와 메모리 상태가 초기화되는지
- 전날 비동기 결과가 오늘 상태를 복원하지 못하는지
- 인증·조회 실패가 마지막 성공 타일을 지우거나 저장 상태를 덮어쓰지 않는지
- 출결 반영 확인 이후에만 완료 타일 강조와 완료 스낵바가 나타나는지
- 기존 알림 시작 흐름과 확인 정책이 유지되는지

전체 검증은 `dart format`, `flutter analyze`, `flutter test`,
`flutter build apk --release --split-per-abi`를 사용한다. 릴리스 APK의 패키지,
증가한 버전 코드와 기존 릴리스 인증서를 확인한 뒤 연결된 Android 실기기에
업데이트 설치한다. 실제 출결 요청은 검증 목적으로 임의 전송하지 않는다.

## 구현 순서

1. 당일 표시 상태와 저장소의 실패 테스트를 추가한다.
2. 날짜·4개 시각·마지막 조회 시각만 저장하는 출결 상태 저장소를 구현한다.
3. 컨트롤러에 항상 존재하는 표시 상태와 진행 목적을 추가한다.
4. 새로고침, 토큰 재사용, 인증 콜백과 출결 동작 흐름을 연결한다.
5. 인증 카드를 제거하고 오늘 출결 카드의 `확인 전` 및 새로고침 UI를 구현한다.
6. 기존 일정 알림 흐름을 공통 목적 처리에 연결한다.
7. 날짜·동시성·오류 회귀 테스트와 문서를 갱신한다.
8. 전체 검증과 Android 릴리스 빌드·실기기 업데이트 설치를 수행한다.
