# 알림 재예약 일관성 개선 계획

## 목적

일정 저장과 OS 알림 재예약이 동시에 여러 번 발생하더라도 최종 알림이 가장 최근에 저장된
일정과 일치하도록 한다. 재예약 도중 앱이 종료되어 오래된 알림이 남더라도, 그 알림이 잘못된
인증·출결 흐름을 시작하지 않도록 한다.

## 현재 알고리즘

현재 일정이 변경되거나 앱이 시작되면 다음 과정을 각각 독립적으로 실행한다.

```text
현재 일정 목록 읽기
→ 기존 OS 알림 전체 삭제
→ 향후 발생 건 계산
→ 가까운 순서대로 최대 60건 등록
```

이 방식의 문제는 재예약 요청마다 위 과정을 바로 시작한다는 점이다.

```text
이전 일정 기준 재예약 A 진행 중
→ 사용자가 일정 수정·삭제
→ 최신 일정 기준 재예약 B 시작
→ A와 B가 동시에 OS 알림을 삭제·등록
→ 먼저 시작한 A가 나중에 끝나면 과거 알림이 최종 상태에 남을 수 있음
```

사용자가 만든 일정끼리 시간이 겹쳐서 발생하는 문제가 아니다. 서로 다른 시각의 일정이라도
두 번의 재예약 작업이 동시에 실행되면 발생할 수 있다.

## 변경할 핵심 원칙

### 1. 저장된 일정을 기준 상태로 사용

사용자가 저장·수정·삭제를 누르면 일정 데이터는 즉시 로컬 저장소와 화면에 반영한다.
OS 알림 등록 결과를 일정 데이터의 기준으로 사용하지 않는다.

```text
사용자 변경
→ 로컬 일정 즉시 저장
→ OS 알림 갱신 요청
```

따라서 알림 재예약이 오래 걸리거나 앱이 중간에 종료되어도 사용자의 일정 변경 자체는
유실되지 않는다.

### 2. OS 알림 재예약은 한 번에 하나만 실행

전체 알림을 삭제하고 다시 등록하는 작업은 동시에 두 개 이상 실행하지 않는다. 현재 작업이
끝나기 전에는 다음 재예약 작업이 OS 알림 시스템에 진입하지 못하게 한다.

```text
대기 상태
→ 재예약 요청 수신
→ 재예약 실행 중
→ 완료
→ 대기 상태
```

이 원칙으로 두 작업이 서로의 알림을 삭제하거나 덮어쓰는 상황을 없앤다.

### 3. 진행 중 들어온 변경은 작업별로 쌓지 않고 하나로 합침

재예약 중 일정이 변경되면 새 작업을 즉시 시작하지 않는다. 대신 “최신 일정으로 다시
재예약해야 함” 상태만 기록한다.

```text
재예약 A 실행 중
→ 일정 변경: 재예약 필요 표시
→ 다시 일정 변경: 재예약 필요 상태 유지
→ 다시 일정 변경: 재예약 필요 상태 유지
→ A 완료
→ 현재 저장된 최신 일정으로 재예약 B 한 번 실행
```

예를 들어 퇴실 시간을 `18:00 → 18:10 → 18:20 → 18:30`으로 연속 수정해도 중간 값마다
재예약하지 않는다. 진행 중인 작업이 끝나면 최종 값인 `18:30` 기준으로 한 번만 재예약한다.

### 4. 각 재예약은 시작 시점의 일정 복사본을 사용

재예약이 시작되면 그 시점의 일정 목록을 별도 복사본으로 고정한다. 실행 도중 화면의 일정
목록이 변경되어도 현재 재예약 계산에는 섞이지 않는다.

```text
재예약 A 시작: 일정 버전 A의 복사본 사용
→ 사용자가 일정 변경
→ A는 처음 받은 복사본으로 마무리
→ 다음 재예약 B가 최신 일정 복사본 사용
```

현재 작업의 입력이 중간에 바뀌는 것을 막고, 이후 최신 상태로 한 번 더 맞추는 방식이다.

## 상태 전이 알고리즘

재예약 관리 상태는 다음 세 가지 개념으로 정리한다.

- `대기`: 실행 중인 재예약이 없음
- `실행 중`: 일정 복사본 하나를 OS 알림에 반영 중
- `재실행 필요`: 실행 중 일정이 변경되어 완료 후 최신 상태로 다시 맞춰야 함

상태 전이는 다음과 같다.

```text
[대기]
  재예약 요청
    → 최신 일정 복사본 생성
    → [실행 중]

[실행 중]
  일정 변경 또는 추가 재예약 요청
    → [재실행 필요] 표시
    → 현재 작업은 계속 진행

[실행 중] 완료
  재실행 필요 없음
    → [대기]

[실행 중] 완료
  재실행 필요 있음
    → 표시 해제
    → 가장 최신 일정 복사본 생성
    → 다음 재예약 실행
```

기존 플랫폼 예약 호출을 중간에 강제로 취소하려 하지 않는 이유는, 이미 Android/iOS로 전달된
알림 등록 호출까지 안전하게 취소된다고 보장하기 어렵기 때문이다. 기존 작업을 끝내고 최신
상태로 한 번 더 덮어쓰는 편이 단순하고 결과가 예측 가능하다.

## 앱이 재예약 중 종료되는 경우

앱 저장소와 OS 알림 시스템을 하나의 원자적 작업으로 묶을 수는 없다. 따라서 재예약 도중 앱이
종료되면 일시적으로 다음 상태가 될 수 있다.

```text
로컬 일정: 최신 상태
OS 알림: 이전 상태 또는 일부만 갱신된 상태
```

이 경우 다음 원칙으로 복구한다.

### 1. 다음 앱 실행 시 전체 재예약

앱 시작 시 로컬에 저장된 최신 일정으로 기존과 같이 전체 재예약한다. 따라서 앱을 다시 열면
OS 알림은 최신 일정으로 수렴한다.

### 2. 알림 실행 시 현재 일정과 재검증

앱을 다시 열기 전에 오래된 알림이 울릴 수 있으므로, 사용자가 알림을 누르면 알림에 들어 있는
정보를 바로 신뢰하지 않는다. 다음 항목을 현재 저장된 일정과 비교한다.

- 같은 일정 ID가 현재도 존재하는가
- 일정이 활성화되어 있는가
- 동작 종류가 같은가
- 날짜와 시각이 같은가
- 현재 공휴일·과정 기간 규칙에서도 실제 발생하는 일정인가

하나라도 다르면 변경·삭제된 과거 알림으로 판단하고 인증을 시작하지 않는다.

```text
알림 누름
→ 현재 일정 로딩 완료 대기
→ 알림 정보와 현재 일정 비교
→ 일치: 기존 인증·출결 흐름 진행
→ 불일치: 과거 알림 안내 후 종료
```

이 방식에서는 오래된 알림이 잠시 화면에 표시될 가능성은 남지만, 그 알림이 잘못된 출결
동작으로 이어지는 것은 차단한다.

## 처리 중 알림을 누르는 경우

출결 상태 조회나 다른 인증 처리가 진행 중일 때 새 알림을 누르면 해당 알림 정보를 즉시
버리지 않는다.

```text
알림 누름
→ 앱이 처리 중이면 알림 정보 보존
→ 현재 처리 완료
→ 저장된 최신 일정과 알림 정보 대조
→ 유효하면 인증 시작, 과거 알림이면 무시
```

한 번에 하나의 예약 출결 흐름만 진행하며, 다음 알림은 현재 흐름이 끝난 뒤 처리한다.

## 실패 처리 원칙

OS 알림 재예약이 실패해도 이미 저장된 일정은 되돌리지 않는다. 사용자 일정과 OS 알림을 함께
되돌리면 사용자가 저장한 변경을 잃을 수 있기 때문이다.

```text
일정 저장 성공
→ OS 알림 재예약 실패
→ 일정은 최신 상태로 유지
→ 알림 예약 실패 상태 표시
→ 다음 앱 실행 또는 다음 일정 변경 때 전체 재예약 재시도
```

무한 자동 재시도는 하지 않는다. 권한 거부나 플랫폼 오류가 지속되는 경우 반복 실행과 배터리
소모를 만들 수 있기 때문이다.

## 변경 후 보장되는 결과

- 전체 재예약 작업이 동시에 실행되지 않는다.
- 재예약 중 여러 번 수정해도 마지막 일정 상태로 한 번만 추가 재예약한다.
- 화면과 로컬 저장소에는 일정 변경이 즉시 반영된다.
- 앱이 계속 실행된다면 OS 알림은 최종적으로 최신 일정과 일치한다.
- 앱이 중간에 종료되어도 다음 실행에서 최신 일정으로 복구한다.
- 종료 전에 남은 과거 알림을 눌러도 현재 일정과 다르면 인증·출결 흐름을 시작하지 않는다.

## 이번 변경에 포함하지 않는 것

- 알림 한 건이 실행될 때 61번째 알림을 추가하는 순환 예약
- 최대 60건 제한 변경
- WorkManager나 포그라운드 서비스 도입
- 일정별 증분 추가·수정·삭제 알고리즘으로의 전환
- SharedPreferences와 OS 알림을 하나의 트랜잭션으로 묶는 시도

현재의 “전체 삭제 후 최신 60건 재등록” 방식은 유지한다. 이번 변경의 목적은 그 전체 재예약이
겹치지 않도록 순서를 보장하고, 중단되더라도 최신 저장 상태로 복구되게 만드는 것이다.

## 확인할 시나리오

1. 앱 시작 재예약 중 퇴실 시간을 수정하면, 최종 OS 알림은 수정된 시각만 사용한다.
2. 앱 시작 재예약 중 일정을 삭제하면, 최종 OS 알림에 삭제된 일정이 남지 않는다.
3. 재예약 중 같은 일정을 여러 번 수정하면, 중간 값별 작업 없이 마지막 상태로 한 번만 다시 예약한다.
4. 재예약 중 앱을 종료한 뒤 다시 실행하면, 저장된 최신 일정으로 전체 알림을 복구한다.
5. 앱 종료 전에 남은 삭제·수정 전 알림을 누르면 인증을 시작하지 않는다.
6. 출결 상태 조회 중 다른 알림을 누르면, 현재 처리 종료 후 해당 알림을 다시 검증하고 처리한다.
