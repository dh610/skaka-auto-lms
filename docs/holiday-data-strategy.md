# 공휴일 데이터 운영 전략

## 목적

이 문서는 공휴일 정보를 실행 시점마다 공공 API에서 조회하지 않고 앱에 정적
데이터로 포함한 이유와 그에 따른 한계, 갱신 기준을 기록한다.

이 선택은 공공 API를 사용할 수 없어서가 아니라 현재 앱의 범위와 운영 조건을
고려한 설계 결정이다.

## 현재 요구사항

- SKALA 과정 기간은 `2026-07-14`부터 `2026-12-18`까지로 고정되어 있다.
- 요일 반복 일정은 사용자가 선택한 경우 법정공휴일에 알림을 만들지 않는다.
- 날짜 지정 일정은 공휴일이어도 사용자가 명시적으로 만든 일정이므로 유지한다.
- 앱은 별도 서버 없이 Android와 iOS 기기에서 동작하는 구성을 지향한다.

현재 필요한 데이터의 범위가 짧고 고정되어 있으므로 앱에는 과정 기간과 겹치는
2026년 공휴일만 포함한다. 실제 데이터는
`lib/features/schedule/domain/training_calendar.dart`에서 관리한다.

## 공공 API를 앱에서 직접 호출하지 않는 이유

### 인증키를 안전하게 숨길 수 없음

모바일 앱에 포함한 문자열과 설정은 난독화 여부와 관계없이 최종 APK 또는 IPA에서
추출될 수 있다. 따라서 공공 API 인증키를 Dart 코드, Android 리소스, iOS plist,
환경 파일 등에 넣어 배포하는 것은 비밀 보관 방법이 아니다.

키가 노출되면 제3자가 할당량을 소진하거나 해당 키로 요청을 보낼 수 있다. 저장소에서
키 파일을 제외하는 `.gitignore`는 Git 유출은 막지만 배포 바이너리에서의 추출은
막지 못한다.

### 안전한 실시간 조회에는 중계 계층이 필요함

인증키를 숨기려면 앱이 직접 공공 API를 호출하는 대신 서버 또는 서버리스 함수가
키를 보관하고 필요한 공휴일 데이터만 앱에 전달해야 한다. 이 경우 다음 운영 요소가
추가된다.

- 배포 및 장애 대응
- 호출량 제한과 오용 방지
- 캐시 및 만료 정책
- 비용과 서비스 수명 관리
- 중계 API 자체의 인증 또는 접근 제어

범위가 2026년 하반기로 고정된 소규모 앱에는 이 운영 비용이 얻는 이점보다 크다고
판단했다.

### 실행 시점의 네트워크 의존성을 줄일 수 있음

공휴일 제외는 일정과 알림을 계획할 때 반드시 필요한 정보다. 정적 데이터는 인터넷
연결, 공공 API 장애, 인증키 상태 및 호출 한도와 관계없이 동일한 결과를 제공한다.
앱 시작 시 외부 조회를 기다릴 필요도 없다.

## 채택한 방식

1. 공식 공공 API에서 과정 기간에 해당하는 공휴일을 개발 시점에 조회한다.
2. 날짜와 명칭을 검토한 뒤 코드의 정적 목록으로 저장한다.
3. 일정 계산과 화면 표시는 동일한 목록을 단일 기준으로 사용한다.
4. 자동 테스트에서 주요 날짜와 공휴일 제외 동작을 검증한다.
5. 공공 API 인증키는 앱과 저장소에 포함하지 않는다.

즉 공공 API는 배포 앱의 런타임 의존성이 아니라 정적 데이터의 출처 및 갱신 도구로
사용한다.

## 장점과 한계

장점:

- 인증키가 배포 바이너리에 포함되지 않는다.
- 별도 서버와 운영 비용이 없다.
- 오프라인에서도 일정 계산 결과가 일관적이다.
- 고정된 과정 기간에서는 구현과 검증이 단순하다.

한계:

- 임시공휴일 지정이나 법령 변경이 앱에 자동 반영되지 않는다.
- 과정 기간 또는 사용 연도가 늘어나면 데이터를 추가하고 앱을 업데이트해야 한다.
- 앱이 범용 서비스로 확대되면 정적 목록의 유지보수 비용이 커진다.

## 재검토 조건

다음 중 하나가 발생하면 서버리스 중계 API 또는 빌드 시 데이터 생성 방식으로 전환을
검토한다.

- 2026년 이후에도 앱을 계속 운영하는 경우
- 여러 기수와 여러 연도의 일정을 지원하는 경우
- 임시공휴일을 앱 업데이트 없이 즉시 반영해야 하는 경우
- 공휴일 외에 지역별 휴일이나 기관별 휴무일까지 관리하는 경우

전환하더라도 앱이 공공 API 인증키를 직접 보유하는 구조는 사용하지 않는다.
