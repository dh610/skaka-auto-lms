# 현재 구현 상태

## 문서 목적

이 문서는 2026-07-24 기준 애플리케이션에 실제로 구현된 범위와 아직 구현하지 않은
범위를 설명한다. 개발 초기의 가설과 조사 과정은
[초기 조사 및 구현 계획](implementation-notes.md)에 별도로 보존한다.

현재 제품의 개발·검증·배포 대상은 Android다. iOS 코드는 공통 Flutter 구현과 기존
시뮬레이터 검증 결과를 보존하기 위해 유지하지만, Apple Developer Program 없이
교육생에게 지속적으로 배포할 수 없고 인증 콜백도 완성되지 않았으므로 iOS 신규 개발과
배포는 중단한다.

## 제품 동작 원칙

이 앱은 화면 잠금 상태에서 Google 인증과 출결 요청을 완전히 무인 실행하는 도구가
아니다. 사용자가 만든 일정에 맞춰 로컬 알림을 표시하고, 알림을 누르면 필요한 경우 Google
인증을 시작한다. Android는 인증 완료 또는 같은 앱 실행 중 유효한 인증 정보가 있을 때 서버의
현재 출결 상태를 확인한 다음 가능한 동작만
처리한다. 보존된 iOS 구현에서는 앱 내 Safari 화면에서 사용자가 직접 처리한다.

```text
사용자 일정 저장
  → OS 로컬 알림 예약
  → 사용자가 알림 선택
  → 필요하면 Chrome Custom Tab에서 Google 인증
  → Android App Link로 앱 복귀
  → 당일 상태 조회
  → 동작별 확인 정책 적용
  → 출결 요청
  → 상태 재조회로 반영 확인
```

## 구현된 기능

- Android와 iOS의 사용자 표시 이름을 `SKALA 출결 도우미`로 통일
- Android와 iOS에 공통 출결 도우미 앱 아이콘 적용

### 사용자 설정

- 최초 실행 시 이름, 캠퍼스 및 반 입력
- 최초 사용자 정보 저장 후 일반 알림, Android 정확 알람, 전체 화면 알람 및 Android 인증 복귀 설정 상태 안내
- 초기 설정의 각 항목을 직접 눌러 해당 권한 또는 Android 설정으로 이동
- Android 통합 설정은 알림 권한 완료를 확인한 뒤 인증 복귀 안내를 표시하고, 사용자가 안내창의 버튼을 눌러야 링크 설정으로 이동
- iOS에서 알림 권한을 이미 거부한 경우 설정 경로를 안내한 뒤 앱 설정으로 이동하고, 권한 허용 후 복귀하면 설정 상태를 다시 확인
- Android 인증 복귀 설정 전에 지원 링크, 지원되는 웹 주소 및 `att.skala-ai.com` 활성화 순서 안내
- 완료 후 일반 알림, Android 정확 알람, 전체 화면 알람 또는 Android 인증 복귀 링크가 해제되면 저장된 완료 상태를 해제하고 앱 복귀·재실행 시 초기 설정을 다시 표시
- 필수 설정을 건너뛰는 동작을 제공하지 않으며 Android에서는 네 설정을 모두 완료해야 앱 사용 가능
- 필요한 설정이 모두 허용된 경우 저장된 최신 일정의 전체 알림 재예약까지 성공해야 초기 설정 화면 통과
- 권한을 해제했다가 다시 허용한 경우 전체 알림을 즉시 재예약하고, 실패하면 초기 설정 화면에서 오류와 재시도 동작 제공
- 판교캠퍼스 4F는 1~5반, 5F는 6~10반으로 선택 제한
- 최초 저장과 정보 수정 시 SKALA 사전 확인 API로 이름·캠퍼스·반 조합 검증
- 검증 실패 시 저장하지 않고 입력 정보 오류와 네트워크·서버 오류를 구분해 안내
- 검증 성공 응답의 사전 인증 토큰은 저장하거나 이후 인증에 재사용하지 않음
- 사용자 정보 로컬 저장 및 수정
- 사용자 정보가 바뀌면 메모리의 인증 토큰, 조회 상태 및 당일 출결 표시 캐시 초기화

### 화면 테마

- 홈 화면 우측 상단에서 시스템 설정, 라이트 모드, 다크 모드 선택
- 선택한 테마를 로컬에 저장하고 앱 재실행 시 복원

### 일정과 공휴일

- 입실, 퇴실, 외출, 복귀 일정 지원
- 요일 반복 일정과 특정 날짜 1회 일정 지원
- 일정 생성, 수정, 활성화, 비활성화 및 삭제
- 비활성화한 일정 카드도 눌러 편집 가능
- 비활성화한 반복 일정은 다음 유효 발생 한 건을 건너뛰고 이후 발생부터 자동 활성화하도록 선택 가능
- 자동 활성화 기준은 과정 기간, 선택 요일 및 공휴일 제외 설정을 반영하며 사용자가 먼저 다시 켜면 건너뛰기 예약을 해제
- 날짜 지정 일정처럼 이후 발생이 없는 경우 자동 활성화 버튼을 표시하지 않음
- 일정 추가·수정 화면에서는 저장 버튼을 하단에 고정하고, 원래 위치의 저장 버튼이 화면에 나타나면 고정 버튼을 숨김
- 일정 추가·수정 화면의 스크롤 하단 여백에 시스템 안전 영역을 반영해 Android 제스처·3버튼 내비게이션과 iOS 홈 인디케이터가 저장 버튼을 가리지 않도록 처리
- 일정 추가·수정 화면 진입 시 오른쪽 스크롤바를 잠시 표시하고 이후 스크롤할 때 다시 표시
- 과정 기간 중 실제 발생 날짜와 시각이 겹치는 일정은 동작 종류와 활성화 여부에 관계없이 중복 저장 방지
- 반복·날짜 지정 일정 사이의 충돌과 공휴일 제외 여부를 함께 계산하고 충돌 날짜 및 기존 동작 안내
- 과정 기간 `2026-07-14`~`2026-12-18` 적용
- 요일 반복 일정에서 2026년 과정 기간 공휴일 제외
- 현재 선택한 요일과 겹쳐 실제로 제외되는 공휴일 표시
- 날짜 지정 일정은 공휴일에도 유지
- 일정별 알람음, 음량, 진동, 점진적 음량 및 다시 알림 간격·최대 횟수 설정
- 실행 시각은 오전·오후 2개 항목과 무한 순환하는 시·분 휠로 선택하며 시·분 숫자를 누르면 현재 선택창 안에서 키패드로 바로 입력 가능
- 키패드를 내리면 유효한 입력을 반영하고 자동으로 시·분 휠로 복귀
- 시 휠이 정오나 자정을 지나면 오전·오후가 자동 전환
- 홈의 `오늘의 알람`에는 오늘 울릴 일정의 시각과 출결 동작만 표시
- 다음 발생 한 건을 건너뛰도록 설정한 일정은 해당 날짜의 `오늘의 알람`에서 제외
- 알림에서 인증 또는 브라우저 흐름을 시작한 발생 기록은 payload 중복 처리 방지를 위해 내부적으로 당일 보존하되 홈에는 상태 배지를 표시하지 않음
- 일정 ID와 예정 발생 시각을 함께 저장해 같은 동작의 서로 다른 발생 건이 충돌하지 않도록 처리
- 완료 기록은 앱 재실행 후에도 당일 유지하며 날짜가 바뀌면 판정에서 제외하고 저장 데이터에서도 정리

공휴일 데이터 선택 근거는 [공휴일 데이터 운영 전략](holiday-data-strategy.md)에
기록되어 있다.

### 알림

- Android와 iOS 로컬 알림 권한 요청
- 플랫폼별 권한 상태를 확인하고 설정이 필요한 경우에만 안내 카드 표시
- Android는 정확 알람 권한으로 각 발생 건을 `AlarmManager`의 알람 시계로 등록
- Android는 전체 화면 알람 권한을 필수로 확인하고 설정 화면에서 상태와 이동 경로 제공
- Android 알람 시각에는 잠금 화면 위 전체 화면을 표시하고 알람음·진동을 반복
- Android 전체 화면의 밀기 동작으로 Google 인증 후 출결 확인 시작
- 다시 알림을 선택하면 다음 알람 시각까지 알림창에 상태를 유지하고 출결 확인 또는 다시 알림 해제 제공
- 최대 다시 알림 횟수가 2회 이상 또는 무제한이고 남은 횟수가 있으면 Android 음량 버튼으로 설정한 간격의 다시 알림 실행
- 다시 알림을 사용할 수 없으면 Android 음량 버튼은 알람 화면을 유지한 채 알람음과 진동만 끔
- 다른 앱을 사용 중일 때 Android가 전체 화면 전환을 제한하면 높은 우선순위 알람 알림으로 표시
- iOS 보존 코드는 기존 로컬 알림 방식을 유지
- 앱 시작, 일정 저장·수정·삭제·활성화 변경 시 향후 알림 재계획
- 알림 재계획은 한 번에 하나만 실행하고, 진행 중 일정이 변경되면 완료 후 최신 일정으로 한 번 더 재계획
- 일정 변경을 알림 재계획보다 먼저 로컬에 저장하여 앱이 중간에 종료되어도 다음 실행 시 최신 일정으로 복구
- 앱 시작 시 저장된 일정은 즉시 화면에 표시하고 OS 알림 재계획과 권한 확인은 이어서 수행
- 앱 재시작과 알림 탭으로 시작된 경우 모두 payload 처리
- 알림 payload를 현재 저장된 일정의 ID·동작·날짜·시각·활성 상태와 다시 대조하고, 변경되거나 삭제된 일정이면 인증을 시작하지 않음
- 출결 처리나 일정 로딩 중 누른 알림 payload는 버리지 않고 현재 처리 완료 후 다시 검증
- 플랫폼 예약 한도를 고려해 가까운 일정부터 최대 60건 계획

알림에는 일정 ID, 출결 동작, 예정 발생 시각만 들어가며 인증 토큰이나 개인정보는 넣지 않는다.

반복 일정도 OS의 무기한 반복 알람 한 건으로 등록하지 않는다. 과정 종료일까지의 실제
발생 시각을 계산한 뒤 각각 독립된 일회성 알림으로 최대 60건 선등록한다. 재계획할 때는
기존 출결 알림을 모두 취소하고 현재 시각과 최신 일정 기준으로 다시 등록한다. 알림이
울리는 것만으로 다음 알림이 추가되지는 않지만, 알림을 눌러 앱이 시작되거나 사용자가
앱을 다시 열면 재계획된다. 앱을 장기간 전혀 열지 않아 60건이 모두 소진되면 이후
알림이 자동 보충되지 않는 한계가 있다.

### 인증과 Android 출결 처리

- 사용자 정보로 SKALA 사전 확인 API 호출
- Android 12 이상에서 `att.skala-ai.com` 지원 링크 연결 상태 확인
- 연결되지 않은 경우 앱 전용 링크 설정 화면으로 안내하고, 허용 후 앱 복귀 시 인증 자동 재개
- Chrome Custom Tab에서 Google OAuth 시작
- `att.skala-ai.com` Android App Link 콜백 수신
- JWT 형식, 만료 시각, 한국 날짜, 지역 및 반 일치 여부 검증
- 토큰을 디스크에 저장하지 않고 앱 프로세스 메모리에서만 사용하며, 같은 실행 중 한국 날짜·사용자 정보가 유효하면 다음 조회 또는 동작에서 재사용 가능
- 홈에는 날짜·요일을 포함한 `오늘 출결` 카드가 항상 표시됨
- 조회 전 네 출결 항목은 `확인 전`으로 표시하고, 조회 성공 후에는 시각을 `HH:mm`, 서버에 값이 없는 항목을 `없음`으로 표시
- 카드 우측 상단 새로고침은 상태만 조회한다. 유효한 메모리 토큰이 없거나 만료되었으면 먼저 Google 인증을 시작하고, 인증 후 상태만 조회한다
- 입실·퇴실·외출·복귀 동작은 누른 뒤 유효한 인증 정보가 없으면 Google 인증을 시작하고, 최신 상태 조회로 가능 여부를 다시 확인한 뒤 각 동작의 확인 정책에 따라 전송한다
- 조회에 성공하면 한국 날짜, 네 출결 표시 시각 및 조회 시각만 로컬에 보관한다. 토큰·네트워크 허용 여부는 보관하지 않으며, 날짜 또는 사용자 정보가 바뀌면 캐시를 삭제하고 다시 `확인 전`으로 표시한다
- 입실·퇴실 / 외출·복귀 상태를 기본 2×2 타일로 표시하고, 카드 폭이 좁거나
  접근성 글자 크기가 큰 경우에는 1열로 전환
- 정상 네트워크 여부는 허용된 경우 숨기고, 출결 전송이 불가능한 경우에만 경고
- 조회 전에는 네 동작을 모두 제시하되, 누른 동작은 최신 상태 확인 뒤 가능한 경우에만 확인·전송한다. 조회 후에는 현재 서버 상태에서 가능한 동작만 제시·전송한다
- 출결 요청 후 상태를 재조회해 반영 여부 확인
- 서버 상태에서 반영이 확인된 동작만 해당 출결 타일 전체를 잠시 강조하고 완료 안내 표시
- 전송 결과가 불명확한 네트워크 실패는 상태만 재조회하고, 서버의 명시적 거부는
  반영 대기 상태를 해제한 뒤 현재 상태를 다시 조회
- 앱이 열린 채 한국 날짜가 바뀌면 전날 토큰, 상태 및 표시 캐시를 폐기하고 새 인증을 요구
- 인증 시작마다 이전 모바일망·VPN의 keep-alive 연결을 폐기하고 현재 네트워크로 새 연결 사용
- 네트워크 단절, 시간 초과, 인증 만료 및 서버 실패를 사용자용 메시지로 구분
- 인증 실패는 인증을 다시 시작하고, 출결 결과 확인 실패는 동작을 재전송하지 않고 상태만 재조회

2026-07-21 재확인 기준으로 SKALA 서버의 `/.well-known/assetlinks.json` 요청은
`HTTP 403`과 출결 웹 HTML을 반환하며 Digital Asset Links JSON을 제공하지 않는다.
따라서 신규 설치에서 App Link가 자동 검증되지 않는다. 앱은 이를
사용자 대신 활성화할 수 없기 때문에 최초 인증 시 Android 설정 화면을 열어 한 번의
사용자 허용을 요청한다. 이후 인증부터는 설정 안내 없이 바로 진행한다.

SKALA는 요청이 허용된 네트워크에서 왔는지 서버에서 확인한다. Android에서 모든 앱에
적용되는 VPN을 사용하면 Wi-Fi를 바꿔도 출결 앱 요청이 VPN 경로를 통과할 수 있다.
이 경우 앱을 VPN의 앱별 적용 대상에서 제외해야 SKALA Wi-Fi를 직접 사용한다.

현재 동작 가능 조건은 다음과 같다.

| 현재 상태 | 가능한 동작 |
|---|---|
| 입실 전 | 입실 |
| 입실 후, 외출 전 | 외출 또는 퇴실 |
| 외출 후, 복귀 전 | 복귀 |
| 복귀 후 | 퇴실 |
| 퇴실 완료 또는 네트워크 불허 | 없음 |

### 출결 확인 정책

- 입실: 확인창 없이 전송
- 복귀: 확인창 없이 전송
- 외출: 하루 한 번만 가능하다는 경고 후 사용자 확인 시 전송
- 오후 5:50 이전 퇴실: 오후 5:50까지 남은 시·분·초를 큰 카운트다운 영역에 실시간으로 보여준 뒤 사용자 확인 시 전송
- 오후 5:50 이후 퇴실: 확인창 없이 전송

확인창의 카운트다운은 확인창이 열린 동안에만 1초 간격으로 갱신된다. 오후 5:50이 되어도
자동으로 퇴실하지 않으며 사용자가 전송 버튼을 눌러야 한다.

## 플랫폼별 현재 범위

| 기능 | Android | iOS 보존 코드 |
|---|---|---|
| 프로필·일정·공휴일 계산 | 지원 | 지원 |
| 로컬 알림 | 전체 화면 알람 지원 | 기존 알림 코드 포함 |
| 브라우저 인증 시작 | Chrome Custom Tab | 앱 내 Safari 화면 |
| 인증 후 앱 복귀 | 지원 | 미구성 |
| 앱 내 상태 조회·출결 전송 | 지원 | 미지원 |

자세한 원인과 iOS 개발 재개 조건은
[Android와 iOS의 기능 차이](platform-capability-differences.md)를 참고한다.

iOS 열은 저장소에 남아 있는 구현 범위이며 현재 사용자 지원 또는 배포 범위를 뜻하지
않는다.

## 저장 데이터와 보안

기기에 지속해서 저장하는 데이터:

- 이름, 캠퍼스, 반
- 사용자가 만든 일정과 활성화 상태
- 당일 알림에서 인증·브라우저 흐름을 시작한 일정 ID, 예정 발생 시각 및 시작 시각
- 조회에 성공한 당일의 한국 날짜, 네 출결 표시 시각 및 조회 시각

지속해서 저장하지 않는 데이터:

- Google 비밀번호
- Google 또는 브라우저 쿠키
- SKALA 출결 Bearer 토큰

공공 API 인증키 역시 앱에 포함하지 않는다. 인증 토큰과 개인정보 전문을 로그나 알림
payload에 기록하지 않는 것을 원칙으로 한다.

## 구현하지 않은 기능

- Google 계정 자동 선택 및 무인 OAuth
- 화면 잠금 상태에서 출결 API 자동 전송
- Wi-Fi 자동 연결·해제 또는 SSID 관리
- 출결 토큰의 영구 저장 및 다음 실행 재사용
- 백그라운드 재시도 Worker
- 날짜별 출결 성공·실패 이력 화면
- 임시공휴일의 원격 실시간 갱신
- iOS Universal Link 콜백과 앱 내 출결 처리

이 목록은 초기 계획에서 의도적으로 제외했거나 향후 검증이 필요한 범위다. 특히 인증과
백그라운드 동작은 플랫폼 보안 정책을 우회하는 방식으로 구현하지 않는다.

## 검증 현황

- Flutter 정적 분석 통과
- 모델, 일정, 공휴일, 알림 계획, API 및 주요 위젯 흐름 자동 테스트
- Android 실기기에서 Chrome Custom Tab 인증, App Link 복귀, 상태 조회 검증
- Android 실기기에서 예약 알림 탭 후 인증 및 예약 동작 연결 검증
- Android 실기기에서 최초 저장·정보 수정 시 올바른 사용자와 잘못된 사용자 조합 검증
- Android 실기기에서 3버튼 내비게이션 사용 시 일정 추가 화면의 하단 저장 버튼이 시스템 영역에 가리지 않는지 검증
- Galaxy S23에서 동일 릴리스 인증서와 패키지명을 확인한 `versionCode 4009` arm64
  APK를 기존 설치 위에 업데이트하고 사용자 정보·일정 유지 및 새 카드 배치 검증
- Flutter 자동 테스트와 정적 분석 통과
- 인증·상태 조회·출결 처리 실패 시 원인별 안내와 안전한 재시도 동작 제공
- iOS Simulator에서 빌드, 설치, 공통 UI 및 앱 내 Safari 인증 화면 검증
- iOS Simulator에서 최초 저장·정보 수정 시 사용자 조합 검증
- iOS Simulator 검증 결과는 보존하되 실기기 검증과 배포는 중단

실제 출결 동작은 테스트 편의를 위해 임의로 전송하지 않고 올바른 출결 시점과 사용자
의사에 따라 검증한다.
