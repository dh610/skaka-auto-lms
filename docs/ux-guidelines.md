# UX 원칙과 설계 결정

이 문서는 SKALA 출결 도우미를 설계·개선하면서 합의한 UX 원칙을 기록한다.
현재 제공하는 기능 목록은 [현재 구현 상태](current-implementation.md)를 기준으로 하며,
이 문서는 기능을 어떤 방식으로 보여주고 동작시킬지 판단하는 기준으로 사용한다.

## 핵심 원칙

### 사용자가 해야 할 일을 최소화한다

- 알림에서 앱을 열었다면 별도의 중간 버튼을 반복해서 누르게 하지 않고 가능한 다음 단계로 바로 연결한다.
- 최초 한 번만 필요한 권한과 플랫폼 설정은 초기 설정에서 모아 안내한다.
- 이미 완료된 설정 안내나 개발·테스트용 기능은 일반 사용 화면에서 반복 노출하지 않는다.
- 자주 사용하는 핵심 동작은 불필요한 스크롤이나 화면 전환 없이 접근 가능해야 한다.

### 상태와 실제 의미를 일치시킨다

- `완료`는 해당 일정 알림에서 인증 또는 브라우저 흐름을 시작했다는 현재 앱의 판정 의미로 사용한다.
- 수행할 필요가 없었던 일정은 완료로 위장하지 않고 `건너뜀`으로 구분한다.
- 예정 시간이 지났지만 아무 처리도 하지 않은 일정은 `시간 지남`으로 표시한다.
- 상태의 문구, 색상, 아이콘은 서로 구분되어야 하며 색상만으로 의미를 전달하지 않는다.
- 실제 서버 출결 완료와 앱 내부의 일정 상태가 다를 수 있음을 문구와 동작에서 숨기지 않는다.

### 위험도에 비례해 확인을 요구한다

- 입실과 복귀처럼 되돌리기 어려운 위험이 낮은 동작에는 불필요한 확인창을 띄우지 않는다.
- 하루 한 번만 가능한 외출은 제한을 명시하고 실행 전 확인을 받는다.
- 오후 5시 50분 이전 퇴실은 기준 시각과 남은 시·분·초를 크게 보여준 뒤 확인을 받는다.
- 오후 5시 50분 이후 퇴실에는 시간 경고를 반복하지 않는다.
- 실수 가능성이 있는 상태 변경은 즉시 토글하기보다 의미를 설명하고 확인받으며 되돌릴 수 있게 한다.

### 플랫폼 차이를 솔직하게 보여준다

- Android와 iOS가 지원하는 인증, 콜백 및 출결 처리 범위를 같은 것처럼 표현하지 않는다.
- iOS에서는 앱 내 Safari를 통해 사용자가 웹에서 직접 처리한다는 사실을 명확히 안내한다.
- Android 전용 App Link 설정은 iOS 화면에 표시하지 않는다.
- 실제 기기에서 검증되지 않은 플랫폼 동작을 지원되는 기능처럼 안내하지 않는다.
- 개발과 배포가 중단된 iOS 구현은 보존 코드임을 명시하고 현재 지원 범위로 홍보하지 않는다.

## 화면과 상호작용 기준

### 초기 설정과 권한 복구

- 사용자 정보 입력 후 필요한 설정을 한 화면에서 확인할 수 있게 한다.
- 최초 저장과 정보 수정 전에 이름·캠퍼스·반 조합을 검증하고, 실패한 정보는 저장하지 않는다.
- 사용자 정보 불일치와 네트워크·서버 오류를 구분해 사용자가 수정할 대상을 오해하지 않게 한다.
- 설정 카드는 카드 전체를 누를 수 있게 하며 `설정 필요`와 `완료` 상태를 명확히 표시한다.
- 권한이나 Android 인증 복귀 링크가 나중에 해제되면 앱 복귀·재실행 시 초기 설정을 다시 표시한다.
- OS 설정 화면을 열기 전에 사용자가 따라야 할 메뉴 경로와 켜야 할 항목을 안내한다.
- iOS 알림 권한처럼 한 번 거부하면 시스템 팝업을 다시 띄울 수 없는 경우 앱 설정으로 이동시키고, 복귀 후 실제 상태를 다시 확인한다.
- 설정을 나중으로 미룰 수는 있지만 필요한 기능을 사용할 때 다시 안내한다.

### 일정 추가와 수정

- 시간은 한국어 환경에서 익숙한 오전·오후 형식으로 표시한다.
- 기본 입력은 시·분 스크롤이며 숫자를 누르면 같은 선택창 안에서 키패드 입력으로 전환한다.
- 시 휠이 정오나 자정을 지나면 오전·오후를 자동으로 전환한다.
- 시·분 휠은 끝에서 멈추지 않고 자연스럽게 순환한다.
- 키보드를 내리면 유효한 입력을 반영하고 스크롤 선택 화면으로 돌아간다.
- 유효하지 않은 시각은 저장하지 않고 입력 범위를 안내한다.
- 저장은 핵심 동작이므로 원래 저장 버튼이 보이지 않을 때 화면 하단에 고정한다.
- 원래 저장 버튼이 화면에 나타나면 고정 버튼을 숨겨 중복 동작처럼 보이지 않게 한다.
- 스크롤 하단 여백은 고정 수치만 사용하지 않고 OS가 제공하는 안전 영역을 함께 반영해 제스처 핸들, Android 3버튼 내비게이션 및 iOS 홈 인디케이터가 저장 버튼을 가리지 않게 한다.
- 화면 진입 직후 오른쪽 스크롤바를 잠시 보여 아래에 추가 설정이 있음을 알리고, 이후에는 스크롤할 때만 다시 표시한다.

### 오늘 예정된 동작

- 상태는 `예정`, `완료`, `시간 지남`, `건너뜀`으로 구분한다.
- 완료는 긍정적인 초록 계열, 시간 지남은 주의가 필요한 색상, 건너뜀은 중립적인 색상을 사용한다.
- 시간 지남 상태는 사용자가 확인 후 건너뜀으로 표시할 수 있고 다시 되돌릴 수 있어야 한다.
- 반복 일정의 완료·건너뜀 기록은 일정 전체가 아니라 `일정 ID + 발생 날짜·시각` 단위로 저장한다.
- 전날 상태가 다음 날 반복 일정에 이어지지 않아야 하며 새 발생 건은 다시 예정 상태에서 시작한다.
- 사용자에게 내부 예약 건수처럼 해석하기 어려운 구현 세부사항을 핵심 정보처럼 노출하지 않는다.

### 인증, 오류와 재시도

- 인증 전 상태에서는 다음 행동을 바로 이해할 수 있는 문구를 사용한다.
- 오류 메시지는 실패 사실뿐 아니라 네트워크, 권한, 인증 만료 등 사용자가 확인할 수 있는 다음 행동을 제시한다.
- 네트워크 변경 후 재시도에 인위적인 고정 대기를 넣지 않는다.
- 이전 실패 상태나 캐시가 새 재시도를 불필요하게 막지 않도록 실제 현재 상태를 다시 확인한다.
- 실제 출결 요청을 테스트 목적으로 보내지 않으며 사용자 의도 없이 동작하지 않는다.

## 시각적 구성 기준

- 사용자 이름의 첫 글자 프로필 이미지처럼 정보 공간을 줄이면서 실질적 가치가 적은 장식은 사용하지 않는다.
- 캠퍼스, 층, 반처럼 중요한 정보가 줄바꿈으로 잘리지 않도록 정보 밀도와 여백을 조정한다.
- 같은 기능으로 이어지는 버튼과 설명을 중복 배치하지 않는다.
- 일회성 안내 패널은 조건이 충족되면 숨긴다.
- 주요 숫자나 남은 시간은 빠르게 읽을 수 있도록 주변 설명과 시각적 위계를 분리한다.
- 라이트·다크·시스템 테마에서 상태 대비와 가독성을 함께 확인한다.

## 접근성과 피드백

- 누를 수 있는 요소는 충분한 터치 영역과 명확한 레이블을 제공한다.
- 깜빡임처럼 지속적으로 시선을 끄는 애니메이션은 피하고 짧은 노출, 페이드 또는 OS 기본 피드백을 우선한다.
- 색상 외에도 문구와 아이콘을 함께 사용한다.
- 작업 중에는 버튼을 중복 실행하지 못하게 하고 진행 상태를 표시한다.
- 성공, 실패, 취소 및 되돌리기 결과를 사용자가 확인할 수 있게 한다.

## 새 기능 검토 체크리스트

새 기능이나 화면 변경을 검토할 때 다음을 확인한다.

1. 사용자가 현재 상태와 다음 행동을 한 번에 이해할 수 있는가?
2. 최초 한 번만 필요한 절차를 매번 반복하게 만들지는 않았는가?
3. 핵심 동작이 화면 크기, 스크롤 위치 또는 키보드에 가려지지 않는가?
4. 실수의 영향이 큰 동작에만 적절한 확인 절차가 있는가?
5. 상태 이름이 실제로 확인한 사실보다 강한 의미를 암시하지 않는가?
6. 오류 후 사용자가 스스로 복구할 수 있는 안내와 재시도 경로가 있는가?
7. Android와 iOS의 차이가 코드와 문구에 명시되어 있는가?
8. 라이트·다크 테마, 작은 화면, 큰 글자, 제스처·버튼 내비게이션 및 권한 해제 후 복귀를 고려했는가?
9. 같은 목적의 버튼이나 설명이 중복되어 있지 않은가?
10. 자동 테스트와 실제 플랫폼 검증으로 핵심 흐름을 확인할 수 있는가?

## 문서 유지 원칙

- 구현 범위가 바뀌면 `current-implementation.md`를 먼저 최신화한다.
- UX 원칙이나 반복해서 사용할 판단 기준이 바뀌면 이 문서를 수정한다.
- 과거 조사와 폐기한 접근은 `implementation-notes.md`에 역사적 기록으로 보존한다.
- 플랫폼 제약의 원인이 바뀌면 `platform-capability-differences.md`도 함께 갱신한다.
