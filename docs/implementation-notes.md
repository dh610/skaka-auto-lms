# SKALA 출결 앱 초기 조사 및 구현 계획

> 문서 상태: 역사적 조사 기록
>
> 조사 기준일: 2026-07-16
>
> 아래 내용은 개발 시작 시점의 가설, 목표와 미검증 항목을 보존한 것이다. 현재 구현을
> 설명하는 문서가 아니며, 완료 여부와 변경된 결정은
> [현재 구현 상태](current-implementation.md)를 참고한다.

## 현재 문서와의 관계

이 문서의 API 관찰 결과와 당시 고민을 추적할 수 있도록 원문 내용은 삭제하지 않는다.
다만 개발 과정에서 다음과 같이 방향이 달라졌다.

- 고정된 09:05/17:55 완전 자동화 대신 사용자가 직접 요일·날짜·시각·동작을 정한다.
- 백그라운드에서 출결 API를 무인 실행하지 않고 로컬 알림을 인증 진입점으로 사용한다.
- Google OAuth는 매번 사용자가 완료한다. 현재 Android는 일반 Chrome 탭 대신 Chrome Custom Tab을 사용한다.
- Android는 인증 콜백 후 상태를 조회하고 허용된 네 종류의 출결 동작을 전송할 수 있다.
- 현재 구현은 인증·상태 조회 오류를 구분하고, 출결 결과가 불확실할 때 동작을 재전송하지 않고 상태만 다시 조회한다.
- 입실과 복귀는 별도 확인 없이 처리하며, 외출과 17:50 이전 퇴실에는 경고를 표시한다.
- Wi-Fi 자동 연결, 토큰 영구 저장, 재시도 Worker 및 출결 이력 화면은 구현하지 않았다.
- 공휴일은 과정 기간이 고정된 점을 고려해 검증된 정적 목록으로 관리한다.

관련 결정은 [공휴일 데이터 운영 전략](holiday-data-strategy.md)과
[Android와 iOS의 기능 차이](platform-capability-differences.md)에 기록한다.

## 1. 문서 목적

이 문서는 Android 개인용 앱에서 SKALA 출결 절차를 자동화하기 위해 실제 모바일 웹 흐름에서 관찰한 API 구조, 인증 방식, 출결 이벤트 요청 방식과 앞으로 검증해야 할 사항을 정리한다.

목표 일정은 다음과 같다.

- 평일 오전 09:05: 입실(`CHECK_IN`)
- 평일 오후 17:55: 퇴실(`CHECK_OUT`)

앱은 반드시 지정된 출석 네트워크에 연결된 경우에만 요청을 시도하며, 성공 여부를 서버 응답으로 확인해야 한다. 기관의 출결 및 자동화 관련 규정을 준수하는 것을 전제로 한다.

## 2. 조사 환경과 범위

- 조사일: 2026-07-16
- 대상: Android 모바일 브라우저에서 실행되는 SKALA 출결 웹앱
- 관찰 도메인:
  - `auth.skala-ai.com`: 본인 확인 화면
  - `att.skala-ai.com`: 출결 상태 및 버튼 화면
  - `lms.skala-ai.com`: 인증 및 출결 API
- 조사 방식:
  - Android 무선 ADB 디버깅
  - Chrome DevTools Protocol을 통한 렌더링 DOM 및 네트워크 이벤트 읽기
  - 실제 출결 상태를 변경하는 외출·퇴실 요청은 전송하지 않음

## 4. 확인된 API

### 4.1 본인 정보 사전 확인

```http
POST https://lms.skala-ai.com/api/public/attendance/pre-verify
Content-Type: application/json
```

요청 본문:

```json
{
  "traineeName": "<사용자 이름>",
  "regionId": "<지역 ID>",
  "subGroupId": "<반 ID>"
}
```

확인할 사항:

- 성공 응답 전체 스키마
- 실패 응답과 오류 코드
- 응답에 포함되는 사전 인증 값의 이름과 유효기간
- 현재 네트워크의 외부 IP가 이 단계에서 검증되는지
- 동일 사용자의 반복 호출 제한 여부

### 4.2 Google OAuth 시작

관찰된 리디렉션 흐름:

```text
GET /api/auth/att-verify-oauth-start
  ↓
GET /oauth2/authorization/google
  ↓
Google 로그인 및 동의
  ↓
GET /login/oauth2/code/google
  ↓
https://att.skala-ai.com/att-checkin
```

사전 확인 결과가 OAuth 시작 요청에 전달되며, OAuth 완료 후 출결 화면에서 사용할 토큰이 발급된다.

확인할 사항:

- 기존 Google 및 LMS 세션이 있을 때 사용자 화면 없이 인증이 완료되는지
- OAuth 과정에서 반드시 필요한 쿠키 목록과 각 쿠키의 수명
- Google 계정 선택 화면이 매일 나타나는지
- OAuth 리디렉션 최종 URL에서 토큰이 어떻게 전달되는지
- 앱의 Custom Tab 또는 WebView에서 동일 흐름이 허용되는지
- OAuth 상태값, PKCE, CSRF 검증 방식

### 4.3 오늘 출결 상태 조회

```http
GET https://lms.skala-ai.com/api/trainee/attendance/today
Authorization: Bearer <당일 출결 토큰>
Cache-Control: no-cache, no-store
```

웹앱은 캐시 방지를 위해 현재 시각 기반의 `_t` 쿼리 파라미터를 추가하기도 한다.

관찰된 응답 구조:

```json
{
  "networkAllowed": true,
  "traineeName": "<사용자 이름>",
  "regionId": "<지역 ID>",
  "classCode": "<반 코드>",
  "today": "YYYY-MM-DD",
  "attendance": {
    "status": "<출결 상태>",
    "checkInTime": "<입실 시각 또는 null>",
    "checkOutTime": "<퇴실 시각 또는 null>",
    "earlyLeaveTime": "<외출 시각 또는 null>",
    "returnTime": "<복귀 시각 또는 null>"
  },
  "events": []
}
```

이 API는 자동화 실행 전후에 사용한다.

- 실행 전: 네트워크 허용 여부와 중복 처리 여부 확인
- 실행 후: 이벤트가 실제로 반영되었는지 재확인

### 4.4 출결 이벤트 기록

```http
POST https://lms.skala-ai.com/api/trainee/attendance/today/record
Authorization: Bearer <당일 출결 토큰>
Content-Type: application/json
```

요청 본문:

```json
{
  "eventType": "CHECK_IN"
}
```

확인된 이벤트 종류:

| 이벤트 | 의미 |
|---|---|
| `CHECK_IN` | 입실 |
| `CHECK_OUT` | 퇴실 |
| `EARLY_LEAVE` | 외출 |
| `RETURN_AFTER_EARLY` | 외출 후 복귀 |

웹앱에서 관찰한 버튼 활성화 조건:

- 입실: 아직 `checkInTime`이 없을 때
- 외출: 입실했고 외출 및 퇴실 기록이 없을 때
- 외출 후 복귀: 외출 기록이 있고 복귀 및 퇴실 기록이 없을 때
- 퇴실: 입실 상태이며, 외출했다면 복귀가 완료된 후

자동화 앱은 현재 목표상 `CHECK_IN`과 `CHECK_OUT`만 사용한다.

## 5. 인증 토큰에 대해 확인된 내용

- 출결 API는 `Authorization: Bearer <token>` 헤더를 사용한다.
- 웹앱은 토큰을 브라우저 `localStorage`의 `att_token`에 저장한다.
- 토큰은 JWT 형식이다.
- JWT에는 사용자, 지역, 반, 적용 날짜 및 만료 시각과 관련된 정보가 포함된다.
- 관찰된 토큰은 단기 토큰이며 당일 날짜에 종속된다.
- 웹앱은 토큰의 날짜가 현재 한국 날짜와 다르면 토큰을 제거한다.

따라서 전날 토큰을 다음 날 재사용하는 방식은 사용할 수 없다. 입실과 퇴실 사이에도 토큰이 만료될 수 있으므로 오후 퇴실 전 새로운 토큰 확보가 필요할 가능성이 높다.

보안 원칙:

- 사용자 Google 비밀번호를 앱에 저장하지 않는다.
- 토큰, 쿠키 및 Wi-Fi 비밀번호는 Android Keystore 기반 암호화 저장소를 사용한다.
- 토큰 전문을 로그나 알림에 기록하지 않는다.
- 디버그 빌드의 HTTP 로깅에서도 인증 헤더와 쿠키를 마스킹한다.

## 6. 목표 자동화 흐름

### 6.1 오전 09:05 입실

```text
09:05 실행 예약
  ↓
오늘이 실행 대상 평일인지 확인
  ↓
오늘 입실 성공 기록이 이미 있는지 확인
  ↓
출석 Wi-Fi 연결 또는 연결 제안
  ↓
SSID와 LMS 접근 가능 여부 확인
  ↓
사전 확인 및 Google OAuth 수행
  ↓
당일 출결 토큰 확보
  ↓
GET /attendance/today
  ↓
networkAllowed == true 확인
  ↓
checkInTime == null 확인
  ↓
POST eventType=CHECK_IN
  ↓
GET /attendance/today 재조회
  ↓
checkInTime이 생성됐는지 확인
  ↓
성공 기록 및 알림
  ↓
출석 Wi-Fi 연결 해제
```

09:05에 실패하면 출석 인정 기준을 넘기기 전에 짧은 간격으로 제한된 횟수만 재시도하고, 자동 처리되지 않으면 즉시 수동 출석 알림을 표시한다.

### 6.2 오후 17:55 퇴실

```text
17:55 실행 예약
  ↓
오늘 입실 여부와 퇴실 성공 기록 확인
  ↓
출석 Wi-Fi 연결 또는 연결 제안
  ↓
유효한 토큰 확인 또는 새 토큰 확보
  ↓
GET /attendance/today
  ↓
networkAllowed == true 확인
  ↓
checkInTime != null 확인
  ↓
checkOutTime == null 확인
  ↓
외출 상태라면 returnTime 존재 여부 확인
  ↓
POST eventType=CHECK_OUT
  ↓
GET /attendance/today 재조회
  ↓
checkOutTime이 생성됐는지 확인
  ↓
성공 기록 및 알림
  ↓
출석 Wi-Fi 연결 해제
```

17:55 이전에는 퇴실 요청을 보내지 않는다. 실패 시 제한된 횟수만 재시도하며, 처리되지 않으면 즉시 수동 퇴실 알림을 표시한다.

## 7. Android 앱 구성 제안

### 주요 모듈

1. `ScheduleManager`
   - 09:05 입실 및 17:55 퇴실 예약
   - 재부팅 후 일정 복구
   - 평일 및 제외 날짜 처리

2. `WifiCoordinator`
   - 출석 Wi-Fi 연결 상태 감지
   - 앱이 등록한 `WifiNetworkSuggestion` 관리
   - 올바른 SSID 및 네트워크 도달 가능성 확인
   - 처리 후 앱이 등록한 네트워크 제안 제거

3. `AuthCoordinator`
   - 사전 확인 API 호출
   - Google OAuth 흐름 수행
   - 당일 토큰 획득과 갱신
   - 토큰 날짜 및 만료 검증

4. `AttendanceApi`
   - 오늘 출결 상태 조회
   - 입실 및 퇴실 이벤트 기록
   - 요청 전후 상태 검증

5. `AttendanceWorker`
   - 전체 자동화 순서 조정
   - 중복 실행 방지
   - 제한된 재시도와 타임아웃

6. `SecureStore`
   - 사용자 설정과 인증 정보 암호화 저장
   - 토큰 및 쿠키의 로그 노출 방지

7. `NotificationManager`
   - 성공·실패·재로그인 필요 알림
   - 수동 처리 바로가기 제공

### 화면

- 홈: 오늘 입실·퇴실 상태와 다음 실행 시각
- 설정: 이름, 지역, 반, Wi-Fi, 실행 요일과 시간
- 인증: Google 계정 연결 및 세션 상태
- 기록: 날짜별 성공·실패와 실패 원인
- 테스트: 상태 조회만 수행하는 안전한 연결 테스트

## 8. 반드시 추가로 알아내야 할 것

### 우선순위 1: 인증 자동 갱신

- 기존 Google 세션으로 사용자 상호작용 없이 OAuth가 완료되는가?
- 앱 재실행 또는 휴대폰 재부팅 후에도 필요한 세션이 유지되는가?
- 오전과 오후에 각각 별도 OAuth가 필요한가?
- Custom Tab을 보이지 않게 실행하는 것이 가능한가, 허용되는가?
- 공식적으로 지원되는 앱용 OAuth 또는 출결 API가 존재하는가?

완전한 백그라운드 자동화 가능 여부를 결정하는 가장 중요한 항목이다.

### 우선순위 2: 네트워크 연결과 검증

- 출석 Wi-Fi의 보안 유형: 개방형, WPA2/WPA3 Personal, Enterprise
- `WifiNetworkSuggestion`으로 연결 가능한가?
- 최초 사용자 승인 후 재연결이 자동으로 이루어지는가?
- 앱이 제안을 제거하면 실제로 해당 Wi-Fi에서 즉시 연결 해제되는가?
- 서버는 외부 IP, SSID, 위치 중 무엇으로 네트워크를 검증하는가?
- Tailscale VPN이 `networkAllowed` 판정에 영향을 주는가?
- 네트워크 연결 중 다른 앱 트래픽을 제한해야 하는가?

### 우선순위 3: 이벤트 API 오류 처리

- `record` API의 성공 응답 전체 스키마
- 중복 입실 및 중복 퇴실 요청의 상태 코드와 응답
- 허용 네트워크가 아닐 때의 상태 코드와 응답
- 토큰 만료 시 상태 코드
- 출석 가능 시간 밖 요청의 상태 코드
- 서버 장애 및 타임아웃 시 요청이 실제 반영됐는지 확인하는 방법
- 동일 idempotency key 또는 중복 방지 기능이 있는지

### 우선순위 4: 정확한 예약 실행

- 휴대폰 Android 버전 및 제조사 배터리 최적화 정책
- 09:05와 17:55의 정확 실행이 필요한지, 허용 가능한 지연 범위
- Exact Alarm 권한을 사용할 수 있는지
- Doze 상태에서 실행이 얼마나 지연되는지
- 앱 강제 종료 및 재부팅 후 자동 복구 방법
- 배터리 최적화 예외가 필요한지

### 우선순위 5: 달력과 예외 처리

- 주말만 제외하면 되는지
- 공휴일, 휴강일, 휴가 및 결석일 데이터 소스
- 사용자가 당일 자동화를 일시 중지하는 방법
- 외출 중인 상태에서 17:55가 됐을 때 자동 퇴실을 금지하고 알림만 보낼지

## 9. 안전한 검증 순서

1. 현재 상태 조회 API만 호출하는 CLI 테스트
2. 토큰 만료 및 날짜 변경 동작 확인
3. Google OAuth 재인증이 사용자 조작 없이 가능한지 확인
4. 테스트 앱에서 Wi-Fi 연결·해제만 검증
5. 앱에서 상태 조회만 수행
6. 실제 입실 가능 시간에 사용자의 확인 아래 `CHECK_IN` 한 번 검증
7. 실제 퇴실 가능 시간에 사용자의 확인 아래 `CHECK_OUT` 한 번 검증
8. 성공 응답과 재조회 결과가 일치하는지 확인
9. 실패·중복·네트워크 차단 시나리오 검증
10. 충분한 검증 이후 예약 자동화를 활성화

테스트 목적으로 외출 또는 퇴실 이벤트를 임의로 생성하지 않는다.

- 요청 및 응답에서 민감정보 마스킹

## 11. 완료 조건

- 지정 네트워크가 아니면 요청을 보내지 않는다.
- 입실은 09:05 이전에 자동 전송되지 않는다.
- 퇴실은 17:55 이전에 자동 전송되지 않는다.
- 같은 이벤트를 하루에 두 번 기록하지 않는다.
- API 성공 응답만으로 판단하지 않고 상태 조회로 반영 여부를 확인한다.
- 인증 만료, 네트워크 실패 및 서버 오류를 사용자에게 즉시 알린다.
- 토큰, 쿠키, 비밀번호와 개인정보를 평문 로그에 남기지 않는다.
- 자동화 일시 중지 및 수동 출결 전환이 가능하다.
